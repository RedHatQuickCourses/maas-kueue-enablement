== The GPU Queueing Problem

Organizations deploying AI platforms on Kubernetes face a fundamental challenge: expensive GPU resources create resource contention in multi-tenant environments, leading to underutilization, unpredictable performance, and operational overhead.

=== The Multi-Tenant GPU Contention Scenario

Consider a production AI cluster with 4 NVIDIA A100 GPUs shared by three teams:

* **Training Team**: Submits batch training jobs (8-hour duration, 4 GPUs per job)
* **Inference Team A**: Runs customer-facing inference API (2 GPUs required, 24/7 uptime)
* **Inference Team B**: Performs batch embeddings generation (2 GPUs, nightly schedule)

**Monday, 9:00 AM - Resource Contention Event:**

The training team submits a PyTorchJob requesting 4 GPUs for hyperparameter tuning. Kubernetes operates first-come-first-served—the training job immediately consumes all 4 GPUs.

**Cascade of failures:**

1. Inference Team A's deployment scales up to handle morning traffic spike
2. New inference pods request GPUs but receive `Insufficient nvidia.com/gpu` errors
3. Customer API latency spikes: 50ms (normal) → 300ms (degraded)
4. Inference Team B's nightly job fails

5. Platform engineer receives pages from both inference teams
6. Engineer manually drains training job, waits 15 minutes for graceful shutdown
7. Re-schedules inference workloads, verifies API latency recovery
8. Total incident duration: **90 minutes from detection to resolution**

**The core problem**: Without Kueue, Kubernetes operates first-come-first-served. If 10 training jobs requesting GPUs submit simultaneously but only 4 GPUs exist, 6 jobs immediately fail with `Insufficient nvidia.com/gpu` errors. Platform engineers spend 15-20 hours weekly manually scheduling workloads, draining nodes, and coordinating team access.

== Kueue Architecture and Components

Kueue is a Kubernetes-native queueing system that manages quota and job admission for scarce resources like GPUs. It wraps around the standard Kubernetes scheduler, adding a queueing layer that controls when workloads can start.

=== What is Kueue

Kueue provides admission control for Kubernetes workloads by managing quotas, enforcing priorities, and queueing jobs when resources are unavailable. It does not replace kube-scheduler—instead, it decides **IF and WHEN** a job can create pods, while kube-scheduler decides **WHERE** those pods run.

**Kueue vs kube-scheduler relationship:**

----
┌───────────────────────────────────────────────────┐
│  Kueue: The Bouncer (Admission Control)          │
│  - Decides IF and WHEN a job can start           │
│  - Manages quotas across teams                   │
│  - Queues workloads when resources unavailable   │
└───────────────────────────────────────────────────┘
                    ↓ (Admits workload)
┌───────────────────────────────────────────────────┐
│  kube-scheduler: The Seating Algorithm            │
│  - Decides WHERE pods run (which nodes)           │
│  - Filters/scores nodes based on requirements     │
│  - Binds pods to nodes                            │
└───────────────────────────────────────────────────┘
----

**Analogy**: Kueue is the registrar's office deciding if you can enroll in a course. The kube-scheduler is the classroom seating algorithm finding you an empty seat once you're admitted.

[TIP]
====
Kueue does not replace kube-scheduler. It wraps around it, adding a queueing layer for admission control before pods are created. This preserves Kubernetes' mature scheduling logic while adding quota management and prioritization.
====

**How Kueue prevents resource contention:**

Without Kueue, 10 PyTorchJobs requesting GPUs submit simultaneously. Kubernetes admits all 10 immediately. If only 4 GPUs exist, 6 jobs fail with `Insufficient nvidia.com/gpu`. With Kueue, the admission webhook intercepts job creation, creates Workload objects representing resource requests, and queues them. Kueue admits workloads sequentially as GPU quota becomes available—no failures, predictable queueing.

=== Core Components

Kueue uses five core resource types to manage workload admission and quota enforcement:

[cols="1,1,2,1",options="header"]
|===
|Component |Scope |Purpose |Who Creates

|Workload
|Namespaced
|Admission ticket representing resource request
|Kueue (auto-created from jobs)

|LocalQueue
|Namespaced
|Entry point for jobs in a namespace/team
|Platform Engineer

|ClusterQueue
|Cluster-wide
|Resource pool manager with quotas
|Platform Engineer

|ResourceFlavor
|Cluster-wide
|Hardware type definition (A100, T4)
|Platform Engineer

|Cohort
|Cluster-wide
|Group of queues sharing elastic quotas
|Platform Engineer
|===

**Workload Lifecycle States:**

A Workload moves through distinct phases managed by Kueue's controller:

* **Pending**: Created and waiting in LocalQueue (no pods running)
* **Admitted**: ClusterQueue approved resource request, job controller creates pods
* **Running**: Job's pods executing on nodes, consuming quota
* **Finished**: Job completed successfully or failed
* **Suspended**: Preempted by higher-priority workload (pods deleted to free resources)

**Job Submission Workflow:**

----
1. User submits PyTorchJob with label:
   kueue.x-k8s.io/queue-name: team-a-queue
         ↓
2. Kueue webhook intercepts job creation,
   creates Workload object (suspends job)
         ↓
3. Workload enters LocalQueue (team-a-queue)
         ↓
4. LocalQueue forwards request to ClusterQueue
   (gpu-training-queue)
         ↓
5. ClusterQueue checks quota:
   4 GPUs available? Yes (admits), No (queues)
         ↓
6. Kueue admits Workload within 2-3 seconds
   (unsuspends job)
         ↓
7. PyTorchJob controller creates pods
         ↓
8. kube-scheduler places pods on GPU nodes
----

This workflow ensures jobs only create pods when quota is available, eliminating `Insufficient nvidia.com/gpu` failures and enabling predictable queuing.

=== ResourceFlavor: Hardware Differentiation

ResourceFlavor defines specific hardware types for heterogeneous clusters, enabling workloads to request particular GPU models, memory sizes, or availability zones.

**Purpose**: Differentiate hardware resources in clusters with mixed GPU types (e.g., NVIDIA A100 for training vs T4 for inference).

**Use Case**: Production cluster with NVIDIA A100-40GB for training workloads and T4-16GB for cost-effective inference.

**Example ResourceFlavor:**

[source,yaml]
----
apiVersion: kueue.x-k8s.io/v1beta1
kind: ResourceFlavor
metadata:
  name: a100-gpu                     # <1>
spec:
  nodeLabels:                        # <2>
    gpu-type: nvidia-a100
    gpu-memory: 40gb
  tolerations:                       # <3>
  - key: "nvidia.com/gpu"
    operator: "Exists"
    effect: "NoSchedule"
----
<1> Flavor name referenced by ClusterQueue resources
<2> Node labels to match hardware (set by cluster admin during node provisioning)
<3> Tolerations allow pods to schedule on tainted GPU nodes

**How it works**: When a ClusterQueue references the `a100-gpu` flavor, Kueue ensures admitted workloads' pods include the `gpu-type: nvidia-a100` node selector and the GPU taint toleration. The kube-scheduler then places pods only on A100 nodes.

**Decision Matrix: When to Create Multiple ResourceFlavors**

[cols="1,2,1",options="header"]
|===
|Cluster Type |ResourceFlavor Strategy |Example

|Homogeneous (all identical GPUs)
|Single default flavor
|✅ All nodes have A100-40GB

|Heterogeneous (mixed GPU types)
|Multiple flavors
|✅ A100 (training) + T4 (inference)

|Multi-zone (availability zones)
|Flavor per zone
|⚠️ Advanced: east-a100, west-a100
|===

Create ResourceFlavors based on workload requirements. Training teams need A100-40GB (high compute, 312 TFLOPS FP16). Inference teams can use T4-16GB (cost-effective, 65 TFLOPS FP16).

=== LocalQueue: Namespace Entry Point

LocalQueue is the namespaced entry point for Kueue workloads. It acts as a bridge, funneling workloads from a specific team or namespace into a cluster-wide resource pool managed by a ClusterQueue.

**Purpose**: Team-specific queue enabling multi-tenancy without requiring cluster-level permissions.

**Characteristics:**

* Namespaced resource (created per team/project)
* Users submit jobs specifying `kueue.x-k8s.io/queue-name: team-a-local-queue`
* Forwards resource requests to designated ClusterQueue
* Enables multi-tenancy (teams don't need cluster admin access to see quotas)

**Example LocalQueue:**

[source,yaml]
----
apiVersion: kueue.x-k8s.io/v1beta1
kind: LocalQueue
metadata:
  name: team-a-local-queue           # <1>
  namespace: team-a-project          # <2>
spec:
  clusterQueue: gpu-inference-queue  # <3>
----
<1> LocalQueue name users reference in job YAML (`kueue.x-k8s.io/queue-name` label)
<2> Team's namespace (typically one LocalQueue per namespace)
<3> Points to cluster-wide ClusterQueue managing quotas

**How users interact**: Data scientists submit PyTorchJobs in their namespace with label:

[source,yaml]
----
metadata:
  labels:
    kueue.x-k8s.io/queue-name: team-a-local-queue
----

Kueue's webhook creates a Workload object, places it in the LocalQueue, which forwards it to the ClusterQueue for admission decisions.

=== ClusterQueue: Cluster-Wide Resource Manager

ClusterQueue manages a cluster-wide pool of resources, defining quotas per ResourceFlavor and enforcing fair sharing across multiple LocalQueues.

**Purpose**: Central resource pool manager with quota enforcement and admission policies.

**Characteristics:**

* Cluster-scoped (only cluster admins create ClusterQueues)
* Defines quotas per ResourceFlavor (e.g., maximum 8 A100 GPUs)
* Aggregates demand from multiple LocalQueues
* Enforces fair sharing and borrowing policies (via cohorts)

**Example ClusterQueue:**

[source,yaml]
----
apiVersion: kueue.x-k8s.io/v1beta1
kind: ClusterQueue
metadata:
  name: gpu-training-queue           # <1>
spec:
  namespaceSelector: {}              # <2>
  cohort: gpu-cohort                 # <3>
  resourceGroups:
  - coveredResources: ["cpu", "memory", "nvidia.com/gpu"]  # <4>
    flavors:
    - name: a100-gpu                 # <5>
      resources:
      - name: cpu
        nominalQuota: 32             # <6>
      - name: memory
        nominalQuota: 128Gi
      - name: nvidia.com/gpu
        nominalQuota: 8              # <7>
----
<1> ClusterQueue name (referenced by LocalQueues via `spec.clusterQueue`)
<2> Empty selector allows all namespaces (or use labels for team-specific access)
<3> Optional: Cohort for elastic quota sharing (Chapter 2 covers cohort configuration)
<4> Resource types governed by this queue
<5> ResourceFlavor defining hardware type (from Section 2.3)
<6> CPU quota: 32 cores nominal quota for this queue's workloads
<7> GPU quota: Maximum 8 A100 GPUs consumable by workloads in this queue

**How quota enforcement works**: When a Workload requests 4 GPUs and the ClusterQueue has `nominalQuota: 8` with 6 GPUs already consumed, Kueue admits the workload (6 + 4 = 10 ≤ 8 nominal, but borrowing from cohort allowed). If borrowing is exhausted, the workload queues until resources free.

=== Queue Hierarchy

Kueue's hierarchical architecture enables multi-tenant GPU sharing:

----
┌─────────────────────────────────────────────────┐
│  Cohort: gpu-cohort (elastic quota sharing)     │
│                                                  │
│  ┌──────────────────┐  ┌──────────────────┐    │
│  │ ClusterQueue:    │  │ ClusterQueue:    │    │
│  │ gpu-training     │  │ gpu-inference    │    │
│  │ 8 A100 GPUs      │  │ 8 T4 GPUs        │    │
│  └──────────────────┘  └──────────────────┘    │
│         ↑                      ↑                 │
└─────────┼──────────────────────┼─────────────────┘
          │                      │
  ┌───────┴────────┐     ┌───────┴────────┐
  │ LocalQueue:    │     │ LocalQueue:    │
  │ team-a-local   │     │ team-b-local   │
  │ (namespace: a) │     │ (namespace: b) │
  └────────────────┘     └────────────────┘
          ↑                      ↑
     [PyTorchJob]           [InferenceService]
----

**Workflow:**

1. **User submits job** in namespace with `kueue.x-k8s.io/queue-name: team-a-local`
2. **LocalQueue receives Workload**, forwards to `gpu-training` ClusterQueue
3. **ClusterQueue checks quota**: 8 A100 GPUs nominal, 6 currently used
4. **Admission decision**: Request 2 GPUs → admitted (6 + 2 = 8 ≤ nominal)
5. **Kueue unsuspends job**, PyTorchJob controller creates pods
6. **kube-scheduler places pods** on A100 nodes (via ResourceFlavor node selectors)

[IMPORTANT]
====
A ClusterQueue's `nominalQuota` defines the maximum resources this queue can consume. In a cohort, queues can borrow unused quota from other queues (elastic borrowing). For example, if `gpu-training` needs 10 GPUs but has `nominalQuota: 8`, it can borrow 2 GPUs from `gpu-inference` if idle. Chapter 2 covers cohort configuration in depth.
====

== Enabling Kueue

Enabling Kueue for quota management requires cluster administrator configuration of the Red Hat build of Kueue Operator and OpenShift AI integration.

=== Prerequisites

Before enabling Kueue, verify the following requirements:

* OpenShift 4.19 or later
* Red Hat build of Kueue Operator installed
* cert-manager Operator for Red Hat OpenShift installed
* Cluster administrator privileges
* GPU nodes labeled (for ResourceFlavors with hardware differentiation)

**Verify Kueue Operator is running:**

[source,bash]
----
$ oc get pods -n openshift-kueue-operator
NAME                                            READY   STATUS
kueue-controller-manager-d9fc745df-ph77w        1/1     Running
openshift-kueue-operator-69cfbf45cf-lwtpm       1/1     Running
----

If pods are not running, install the Red Hat build of Kueue Operator from OperatorHub before proceeding.

=== Activate Kueue in OpenShift AI

OpenShift AI integrates with Kueue through the DataScienceCluster Custom Resource. Set `managementState: Unmanaged` to activate the integration.

**Step 1: Patch DataScienceCluster**

[source,bash]
----
$ oc patch datasciencecluster default-dsc \
  --type='merge' \
  -p '{"spec":{"components":{"kueue":{"managementState":"Unmanaged"}}}}' \
  -n redhat-ods-operator
datasciencecluster.datasciencecluster.opendatahub.io/default-dsc patched
----

**What this does:**

* Instructs OpenShift AI to integrate with external Kueue Operator
* Creates default ClusterQueue if none exists (named `default-queue`)
* Enables Kueue integration for InferenceService, Notebook, PyTorchJob, RayJob workloads

**Verification:**

[source,bash]
----
$ oc get clusterqueues
NAME            COHORT   STRATEGY         PENDING WORKLOADS
default-queue   -        BestEffortFIFO   0
----

[WARNING]
====
Do not set `managementState: Managed`. The embedded Kueue is deprecated starting with OpenShift AI 2.24. Always use `Unmanaged` with Red Hat build of Kueue Operator to avoid controller conflicts.
====

=== Enable Namespace for Kueue Management

Kueue validation and queue enforcement apply only to namespaces labeled with `kueue.openshift.io/managed=true`. This label activates the admission webhook for that namespace.

**Step 2: Label namespace**

[source,bash]
----
$ oc label namespace team-a-project kueue.openshift.io/managed=true --overwrite
namespace/team-a-project labeled
----

**What this does:**

* Activates Kueue admission webhook for this namespace
* Enforces `kueue.x-k8s.io/queue-name` label requirement on workloads
* Creates default LocalQueue if none exists

**Verification:**

[source,bash]
----
$ oc get localqueues -n team-a-project
NAME                  CLUSTERQUEUE    PENDING WORKLOADS
default-localqueue    default-queue   0
----

[NOTE]
====
The label `kueue.openshift.io/managed=true` replaces the legacy `kueue-managed` label. Use the new label for forward compatibility with future Kueue releases.
====

=== Configure ClusterQueue with Quotas

ClusterQueues manage resource pools with quotas. Create ClusterQueues for distinct workload types (training, inference) with appropriate quota allocations.

**Step 4: Create ClusterQueue with GPU quotas**

[source,yaml]
----
apiVersion: kueue.x-k8s.io/v1beta1
kind: ClusterQueue
metadata:
  name: gpu-training-queue
spec:
  namespaceSelector: {}
  resourceGroups:
  - coveredResources: ["cpu", "memory", "nvidia.com/gpu"]
    flavors:
    - name: a100-training-gpu              # <1>
      resources:
      - name: cpu
        nominalQuota: 64                   # <2>
      - name: memory
        nominalQuota: 256Gi
      - name: nvidia.com/gpu
        nominalQuota: 8                    # <3>
----
<1> References ResourceFlavor created in Step 3
<2> CPU quota: 64 cores for this queue's workloads
<3> GPU quota: Maximum 8 A100 GPUs consumable by this queue

[IMPORTANT]
====
Set `nominalQuota` based on cluster capacity and team allocation. Example: Cluster with 16 total A100 GPUs → allocate 8 to training queue (`gpu-training-queue`), 8 to inference queue (`gpu-inference-queue`). Adjust based on workload mix and business priorities.
====

**Apply:**

[source,bash]
----
$ oc apply -f gpu-training-clusterqueue.yaml
clusterqueue.kueue.x-k8s.io/gpu-training-queue created
----

== Production Considerations

Production Kueue deployments require strategic decisions about queue architecture, ResourceFlavor configuration, and priority mapping.

=== Queue Strategy Decision Matrix

[cols="2,2,2",options="header"]
|===
|Scenario |Queue Strategy |Rationale

|Single team, homogeneous workloads
|✅ Single LocalQueue
|Simplest: one queue per namespace

|Single team, mixed workloads (training + inference)
|⚠️ Two LocalQueues → different ClusterQueues
|Separate quotas for training vs inference

|Multiple teams, shared GPU pool
|✅ Multiple LocalQueues → single ClusterQueue
|Fair sharing with cohort borrowing

|Multiple teams, dedicated GPUs per team
|⚠️ Multiple LocalQueues → dedicated ClusterQueues
|Hard quota isolation, no borrowing

|Enterprise multi-tenant
|✅ Hierarchical: Cohort → ClusterQueues → LocalQueues
|Elastic sharing + team quotas (Ch2)
|===

[TIP]
====
Start simple: Create one LocalQueue per namespace pointing to a shared ClusterQueue. Add complexity (cohorts, multiple ClusterQueues) only when business requirements demand quota isolation or elastic borrowing. Over-engineering queue hierarchies increases operational complexity.
====

=== ResourceFlavor Strategy

[cols="2,2,2",options="header"]
|===
|Cluster Hardware |ResourceFlavor Strategy |Example Configuration

|All nodes identical (8x A100-40GB)

|1 ResourceFlavor: `default-gpu`

|Mixed GPU types (A100 + T4)

|===

Create ResourceFlavors based on workload requirements. Training teams need A100-40GB (high compute, 312 TFLOPS FP16). Inference teams can use T4-16GB (cost-effective, 65 TFLOPS FP16, 4x lower cost).

== Monitoring and Visibility

Kueue provides command-line tools and Prometheus metrics for monitoring queue status, diagnosing pending workloads, and tracking quota utilization.

=== Queue Status Commands

**Check ClusterQueue status:**

[source,bash]
----
$ oc get clusterqueues
NAME                  COHORT       STRATEGY         PENDING WORKLOADS
gpu-training-queue    gpu-cohort   BestEffortFIFO   3
gpu-inference-queue   gpu-cohort   BestEffortFIFO   0
----

**Interpret output:**

* `PENDING WORKLOADS: 3` means 3 workloads waiting for resources
* `COHORT: gpu-cohort` indicates elastic quota sharing enabled
* `STRATEGY: BestEffortFIFO` means first-in-first-out admission within priority tiers

**Check LocalQueue status:**

[source,bash]
----
$ oc get localqueues -n team-training
NAME                   CLUSTERQUEUE          PENDING WORKLOADS
training-team-queue    gpu-training-queue    1
----

**Diagnose pending workload:**

[source,bash]
----
$ oc get workloads -n team-training
NAME                 QUEUE                  ADMITTED   AGE
pytorch-job-abc123   training-team-queue    False      2m

$ oc describe workload pytorch-job-abc123 -n team-training
...
Status:
  Conditions:
    Message: couldn't assign flavors: insufficient quota for nvidia.com/gpu in flavor a100-gpu
    Reason: Pending
    Type: QuotaReserved
----

**Common Pending Reasons:**

* `insufficient quota for nvidia.com/gpu`: Queue's nominalQuota exhausted (wait for running jobs to finish)
* `couldn't assign flavors`: ResourceFlavor not available (check flavor exists, node labels match)
* Waiting for higher-priority workload to finish (normal queueing behavior)
